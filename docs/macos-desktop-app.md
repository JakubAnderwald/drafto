# Drafto macOS Desktop App — Implementation Plan

## Context

Drafto has a web app (Next.js) and mobile app (Expo + React Native 0.83). The mobile app is offline-first with WatermelonDB (SQLite) and custom pull/push sync with Supabase. The goal is a **native** macOS desktop app with the same offline-first architecture, distributed via Mac App Store. "Native" means real AppKit components — not a webview wrapper like Electron/Tauri.

## Technology Choice: React Native macOS

**Why:** Microsoft's [react-native-macos](https://github.com/microsoft/react-native-macos) renders actual native AppKit components — native text, native sidebar, native context menus. It maximizes code reuse with the existing mobile app while delivering a genuinely native Mac experience. Used by Microsoft in their Office apps.

| Option                 | Verdict    | Reason                                                                                      |
| ---------------------- | ---------- | ------------------------------------------------------------------------------------------- |
| **React Native macOS** | **Chosen** | Native AppKit, ~70% mobile code reuse (WatermelonDB, sync, hooks, providers), same language |
| Tauri v2               | Rejected   | WebView-based — not truly native Mac experience                                             |
| Electron               | Rejected   | WebView-based, 100-150MB bundle, poor Mac feel                                              |
| Swift/SwiftUI          | Rejected   | Best native feel but zero code reuse, entirely separate codebase                            |

### Key Constraints

- **react-native-macos 0.81.5** (latest stable, verified 2026-03-29) — requires React 19 + RN 0.81.6, so no React version gap with mobile
- **No Expo**: Expo doesn't support macOS. Must use bare React Native workflow
- **WatermelonDB macOS**: Not officially supported, but confirmed working via `pnpm patch` — only the podspec platform line needed changing (zero native code changes). `@nozbe/simdjson` also needed the same podspec patch
- **TenTap editor**: Uses react-native-webview internally (WebView for editor area only, rest of app is native). Needs macOS verification; fallback is a direct WKWebView + TipTap bridge

### Code Reuse from Mobile (~70%)

| Layer                           | Reusable | Adaptation needed                                     |
| ------------------------------- | -------- | ----------------------------------------------------- |
| `@drafto/shared`                | 100%     | None                                                  |

## Key Design Decisions

### 1. Database: WatermelonDB (same as mobile)

Reuse WatermelonDB directly. The JSI C++ SQLite adapter is platform-agnostic — only the native module bridge needs a macOS target.

**Confirmed approach (Phase 1):** The iOS native code (`DatabasePlatformIOS.mm`, ObjC bridge) has zero UIKit imports — it uses only Foundation framework (`NSDocumentDirectory`, `NSFileManager`, `NSLog`). The only change needed was adding `:macos => "13.0"` to the `WatermelonDB.podspec` and `simdjson.podspec` platform hashes, done via `pnpm patch`. No code changes.

This gives us **identical** schema, models, migrations, and reactive observable queries as mobile.

### 2. Editor: TenTap (@10play/tentap-editor)

Same as mobile. TenTap uses a WebView internally for the rich text area — this is standard practice even for "native" editors (Bear uses a custom text engine, but most apps use WebKit for rich text). The rest of the app remains fully native.

**Risk**: TenTap hasn't been explicitly tested on macOS. Fallback plan: Create a thin native module that embeds WKWebView with TipTap loaded directly, using the same bridge protocol TenTap uses.

### 3. Auth: Direct Supabase + react-native-keychain

Same pattern as mobile. Session tokens stored in macOS Keychain (via react-native-keychain) instead of Expo Secure Store. Same approval gate (`profiles.is_approved`).

### 4. Sync: Nearly identical to mobile

The sync logic in `apps/mobile/src/db/sync.ts` uses WatermelonDB's `synchronize()` — which works identically on macOS since the database layer is the same. Only the sync triggers change:

- `AppState` (foreground) → `NSApplication.didBecomeActiveNotification`
- Network events via `@react-native-community/netinfo`
- Same 30s periodic sync, same retry with exponential backoff

### 5. macOS-Native UI Patterns

- **Sidebar**: Native macOS sidebar using React Native's `View` with macOS-appropriate styling, or bridged `NSSplitViewController`
- **Toolbar**: Native macOS window toolbar via native module
- **Menu bar**: File > New Note, Edit > Undo/Redo, View > Toggle Sidebar — via native `NSMenu` bridge
- **Keyboard shortcuts**: Cmd+N (new note), Cmd+K (search), Cmd+Shift+Delete (trash)
- **Dark mode**: Automatic from macOS system appearance
- **Window management**: Remember size/position, standard macOS window behaviors

## Mac App Store Deployment

- **Apple Developer Team**: `4J2USPSG2U` (already exists from iOS app)
- **Bundle ID**: `eu.drafto.mobile` (shared with iOS — multi-platform app in App Store Connect)
- **Certificates**: "Apple Distribution" (unified cert for iOS + macOS)
- **Entitlements** (`Drafto.entitlements`):
  - `com.apple.security.app-sandbox = true` (required for MAS)
  - `com.apple.security.network.client = true` (Supabase API)
  - `com.apple.security.files.user-selected.read-write = true` (file attachments)
- **Min macOS**: 14.0 (Sonoma) — required by react-native-macos 0.81.5 framework dependencies
- **Build**: Xcode archive → export for Mac App Store
- **Submit**: `xcrun altool --upload-app --apiKey <key> --apiIssuer <issuer>` or Transporter CLI with JWT auth. Store App Store Connect API keys in CI secrets

## CI/CD

New workflow `.github/workflows/desktop-beta-release.yml`:

1. macOS runner (`macos-14` or `macos-15`)
2. Install Node.js + pnpm + CocoaPods
3. `pnpm install` + `pod install` in `macos/`
4. Build with `xcodebuild` (archive + export)
5. Code sign with certificates from GitHub Secrets
6. Upload to App Store Connect via `xcrun altool --apiKey --apiIssuer` or Transporter CLI with JWT (API keys stored in GitHub Secrets)

Add to `turbo.json` — desktop depends on `@drafto/shared`.

## Testing

### Desktop Tests

- **Unit** (Jest, matching mobile): WatermelonDB models, sync logic, data layer
- **Integration**: Component rendering with mock providers
- **E2E (macOS)**: Detox for macOS (preferred) or AppleScript-based UI automation. Test full flows: login → create notebook → create/edit note → sync → search → trash → restore

### Cross-Platform E2E Tests

New test suite verifying data flows correctly across all 4 platforms. Located in `e2e/cross-platform/` at the repo root:

1. **Sync round-trip tests** — Create note on platform A → verify it appears on platforms B, C, D after sync
   - Desktop → Web, iOS, Android
   - Mobile → Desktop, Web
   - Web → Desktop, Mobile
2. **Content format interop** — Edit rich text on web (BlockNote) → verify renders correctly on mobile/desktop (TipTap) and vice versa
3. **Attachment sync** — Upload attachment on one platform → verify download works on all others
4. **Conflict resolution** — Edit same note on two platforms while offline → verify server-wins resolution on sync

**Implementation approach**: Use a shared test runner (Node.js script) that:

- Controls web via Playwright (existing)
- Controls macOS via Detox or AppleScript
- Controls Android via Maestro (existing) or Detox
- Controls iOS via Detox
- Uses isolated test identities/workspaces per test case (or per platform run)
- Seeds deterministic fixtures before each scenario and cleans up after completion
- Waits for sync intervals between platform actions

**Minimum viable cross-platform E2E**: Start with Desktop ↔ Web sync tests (both can run on CI macOS runner), then expand to include mobile simulators.

## Design System Consistency

All desktop UI must follow the existing design system defined in `apps/web/src/app/globals.css` and the mobile tokens in `apps/mobile/src/theme/tokens.ts`.

**Rules:**

- Use semantic color tokens (bg, fg, fg-muted, border, primary, accent) — never hardcoded colors
- Check `apps/mobile/src/components/ui/` and `apps/web/src/components/ui/` before creating any new component
- When a new macOS-specific UI component is created (e.g., native toolbar wrapper, split view), add it to:
  1. `apps/desktop/src/components/ui/` with variant props and token usage
  2. The design system showcase page (`apps/web/src/app/design-system/page.tsx`) — document the macOS variant alongside web/mobile
- Maintain a mapping file `apps/desktop/src/theme/tokens.ts` that maps the shared design tokens to macOS-native equivalents (NSColor system colors, etc.)
- Any new token added for macOS must also be added to `globals.css` and `apps/mobile/src/theme/tokens.ts` for cross-platform consistency

- react-native-macos-init generates an unused `Drafto-iOS` target — can be cleaned up later
- Min macOS is 14.0 (not 13.0 as originally planned) — required by react-native-macos 0.81.5
